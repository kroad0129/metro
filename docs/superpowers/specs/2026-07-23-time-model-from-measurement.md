# 시간·위치 모델 재설계 — 실측 근거

날짜: 2026-07-23
상태: 구현 완료

## 측정

증미·가양·여의도 3개 역을 10초 간격으로 ~3.7분 관측(`scratchpad/cadence.jsonl`). 핵심 발견:

**`barvlDt`(선택역까지 남은 초)는 실시간 카운트다운이 아니라 "위상별 고정 추정치"다.**

- 전역(1정거장 전)에 있는 열차: **항상 95~100초** (측정 32건 전부, 평균 95).
- 전전역: ~220초. 내역 진입: ~20초. 위상마다 고정값이고, 열차가 다음 위상으로 넘어갈 때만 툭 바뀐다.
- 열차 9075는 등촌(전역)에서 **132초 동안 지연**됐는데 barvlDt는 내내 95로 고정 → 다음 위상(진입)에서 20으로 하강.
- barvlDt가 **위로 튄(지연을 값 증가로 표현한) 사례: 0건.** 지연은 "전역에 오래 얼어있는" 형태로만 나타난다.

## 이전 모델의 문제

barvlDt를 매초 −1로 깎아 부드러운 카운트다운/점 이동을 만들었는데:
- 서버 값이 얼어 있어 폴링마다 되돌아가거나(줄었다 늘었다), 심하면 로컬 카운트다운이 0까지 내려가 **열차가 전역에 서 있는데 "도착"으로 표시**되는 오류.
- 급행 지연도 barvlDt가 안 변하니 반영 불가.

## 새 모델 (구현됨)

barvlDt로 연속 카운트다운을 만드는 건 원천 불가 → **가짜 카운트다운을 버린다.**

1. **점 위치 = 실제 있는 역(arvlMsg3) + 위상 미세보정.** `trainLeftPercent(track, train)`이 현재역 index에 status 보정(진입 −0.35 / 정차 0 / 출발 +0.35 / 운행 0)을 더한다. 시간 미사용. 지연되면 점이 그 역에 그대로 머문다(거짓 도착 없음). 역을 넘을 때만(폴링 시) 이동하고 CSS transition이 부드럽게 잇는다.
2. **시간 = barvlDt를 분 단위 올림(ceil).** `formatRemaining`: 95초 → "2분"(사용자 기대와 일치). 초는 얼어있어 무의미하므로 버린다. 폴링(15초)마다 갱신.
3. **급행 지연은 자동 반영.** 지연된 일반 열차는 "전역에 오래 머무는 점 + 도착"으로 정확히 보인다 — 별도 모델링 없이.

## 제거된 것
- `SECONDS_PER_GAP`, 시간 기반 `trainLeftPercent`, `remainingAt`(브라우저 카운트다운), 단조 보정(`monotonicRemaining` 전체), `ARRIVAL_SOON_SECONDS`. `useNow`는 "N초 전 갱신" 표시용으로만 남는다.

## 트레이드오프
초 단위 실시간 카운트다운(초기 요청 기능)을 포기했다. 실측이 그게 허구임을 보여줬고, 사용자 우선순위가 "정확·불편 없음"으로 바뀌었기 때문. 대신 폴링을 켜면 15초마다 갱신되어 충분히 살아있다.

## 추가 실측 — `recptnDt`와 `ordkey` (모델 재료 교체)

벤더 명세를 정독하고 재측정한 결과, 앞선 결론을 보강하는 사실 셋을 확인했다.

1. **`recptnDt`는 10~27초마다 갱신되는데도 `barvlDt`는 그대로다.** 같은 역에 있는 동안
   3정거장이면 계속 345초. 즉 barvlDt는 "역 단위 조회표"가 맞다(앞선 결론 확정).
2. **그러나 벤더 명세는 명시한다**: "현재시각과 recptnDt의 차이만큼 열차가 더 진행한 것으로
   보정해서 사용해야 합니다." → 카운트다운 기준점은 우리가 받은 시각이 아니라 **recptnDt**다.
3. **`ordkey`에 남은 정거장 수가 정수로 들어 있다** (구조: 상하행1+순번1+거리3+목적지+급행1).
   예 `01003개화0` → 3정거장. 역명 매칭보다 정확하고, 표기 불일치 위험이 없다.

또한 정거장당 소요는 상수가 아니다(실측 125~187초). 그래서 페이스를 하드코딩하지 않고
**`barvlDt ÷ ordkey거리`로 그 열차에서 직접 얻는다.**

## 가상 열차 모델 (2026-07-23 확정, CSS-cap 보간을 대체)

CSS-cap 보간은 트랜지션 시간이 이동거리와 무관해 "홱 튀는" 문제가 있었고, 시간 표시가 폴링
사이 얼어 있었다. 이를 **클라이언트 시뮬레이션(dead reckoning + 관측 보정)**으로 대체했다 —
`src/utils/virtualTrain.ts`.

원리 (실제 지하철 운행 특성을 그대로 모델링). 하드코딩된 페이스 없이 전부 응답에서 유도한다:
- **거리** = `ordkey`의 3자리 정수(`stationsAway`). 없거나 형식이 어긋나면 역 순번 차이로 대체.
- **기준 시각** = `recptnAt`(= `recptnDt`를 KST ISO로 변환). 클라이언트가 앵커를 따로 기록할
  필요가 없어, 폴링을 건너뛰어도 기준이 흔들리지 않는다.
- **시간**(`liveRemainingSeconds`) = `barvlDt − (지금 − recptnAt)`.
- **상한(floor)**: 다음 역에서 API가 보고할 값 `barvlDt × (d−1)/d` 밑으로는 내려가지 않는다
  (실측 검증: 885초/7정거장 → 6정거장 실제 760초, 추정 758초). 마지막 한 정거장은 0이 아니라
  `barvlDt × 0.2`(= 내 역 진입 시점, 실측 95초→20초)에서 멈춘다. 그래서 지연되어도
  카운트다운이 0으로 떨어져 거짓 도착이 되는 일이 없다.
- **위치**(`virtualGaps`) = 시간이 barvlDt에서 floor까지 줄어든 비율만큼 한 정거장 전진.
  위치에도 같은 상한을 걸어(`nextStationGaps`) 마지막 한 정거장은 역에 붙지 않고 코앞에서
  멈춘다 — 진입이 확인(stationsAway=0)돼야 100%에 닿는다.
- **정차(ARRIVED)는 전진하지 않는다** — 서 있는 열차는 움직이지 않아야 하고, 급행 대기 지연이
  "그 역에 서 있는 점"으로 그대로 보인다.
- 시간과 위치가 한 모델에서 나와 모순 불가하고, **매초 초 단위로 줄어든다**("3분 얼어있음" 해결).
  표기는 "1분 50초"처럼 분+초(`formatRemaining`).
- 지나간 열차(gaps < 0)는 그리지 않는다(내 역 출발 열차 유령 문제도 해결).

## 견고화 라운드 (2026-07-23 오후) — 재생 검증 도입

"멈춤·튐"을 사용자 신고가 아니라 기계 검증으로 잡기 위해, 실제 API를 10초 간격 6분(36폴링)
녹화한 픽스처(`__fixtures__/replay-jeungmi.json`)를 실제 모듈에 그대로 통과시키고 1초 단위로
샘플링해 4가지 약속을 단언하는 **재생 테스트**(`virtualTrain.replay.test.ts`)를 만들었다:
시간 절대 안 늘어남 / 점 절대 안 뒤로 감 / 폴링 점프 한도 / 멈춤은 반드시 "지연"으로 감지.

이 검증이 잡아서 고친 결함들:
1. **페이스 학습**(`paceTable`): barvlDt는 (역·방향·거리)별 고정 조회표이므로, 앞차가 d-1에서
   보고한 값을 뒤차의 카운트다운 바닥으로 쓴다(균등 분배는 폴백). localStorage에 역별 저장.
   시발역 대기(개화→김포공항 310초) 같은 구간 특이 소요도 한 번 보면 학습된다.
2. **구간 전이 연속성 클램프**: 균등 바닥이 실제 다음 barvlDt보다 낮으면 전이 때 시간이
   +2초쯤 거꾸로 뛰었다 — 직전 표시값과 이어붙이도록 기준을 백데이트.
3. **정차→출발 재앵커**(`segmentTracker.moveStart*`): 정차 중 시간이 흐른 만큼 출발 순간
   점이 앞으로 확 뛰던 결함 제거. 이동 진행은 "이동 경과시간 ÷ 이동 소요"로 계산하고,
   긴 정차로 시간 예산이 바닥나도 최소 주행시간(40초)을 보장해 순간이동을 막는다.
4. **지연 표시**: 운영사 추정 소요를 20초 이상 초과해 같은 구간에 머물면(`stallSeconds`)
   점 아래 "지연"을 단다. 실제 지연(당산→선유도 추정 125초, 실측 191초)은 어떤 모델도
   없앨 수 없다 — 멈춘 점이 정직한 표시이며, 이 배지가 멈춤을 정보로 바꾼다.
5. **폴링 실패 내성**: 실패해도 마지막 데이터를 유지하고 배너로만 알린다(화면이 통째로
   사라졌다 돌아오는 깜빡임 제거). 한 폴링에서 사라진 열차의 앵커도 90초 보존(수신 흔들림).

남은 한계(정직한 것들): 앱 시작 직후 처음 보는 열차는 구간 내 위치를 몰라 "절반" 가정 —
첫 전이에서 최대 반 구간의 보정이 한 번 온다(페이스 테이블이 localStorage에 쌓이면 축소).
실제 지연 중에는 점·시간이 바닥에 멈춘다 — "지연" 배지가 그 이유를 말해준다.

향후: 열차가 희소하거나 깜빡이면 `realtimePosition`(열차위치정보)으로 연속성 보강.
그 다음 킬러 기능은 "걷는 시간 → 행동 판단"([[upstream-station-query-for-stop-status]]와 별개).
