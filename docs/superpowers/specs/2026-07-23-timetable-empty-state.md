# 심야·막차 안내 — 시간표 기준 다음 출발

날짜: 2026-07-23
상태: 구현 완료

## 문제

실시간 도착 API의 시야는 약 20분이다. 심야·막차 이후·배차가 긴 시간대에는 접근 중인
열차가 하나도 없어 화면이 "접근 중인 열차 없음"만 보여줬다 — 언제 오는지 전혀 알 수 없는
실제 구멍(사용자와 합의된 개선 1순위).

## 데이터 (실측 검증)

- **서울시 역별 시간표 API** `SearchSTNTimeTableByIDService` — 9호선 포함 확인(증미 평일 124+126편).
- 호스트가 실시간 API와 다르다: `openapi.seoul.go.kr:8088` (설정 `seoulTimetableBaseUrl`).
- **사용자의 기존 키가 이 API에서도 작동** 확인 — 추가 키 불필요.
- **역 코드 = 4100 + 역 순번** (개화 4101 ~ 중앙보훈병원 4138, 5개 역으로 검증).
- 시간은 **운행일 기준 표기**: 자정 넘은 막차를 "24:48:25"처럼 24시+로 적는다. 그래서 하루
  시간표 안에서 단조 증가하고, KST 새벽 3시 전까지는 전날 운행일의 24시+로 비교하면 된다.
- WEEK_TAG: 1 평일 / 2 토요일 / 3 휴일. INOUT_TAG(1/2)의 의미는 신뢰하지 않고 **행의
  종착역 코드로 방향을 판정**한다(종착 코드 < 역 코드 → UP). 두 축을 합쳐야 전체다.
- 실측 봉투: 성공 `{ SearchSTNTimeTableByIDService: { RESULT, row } }`, 실패는 최상위 `{ RESULT }`.

## 구현

- **백엔드**: `TimetableService.nextDeparture(station, direction, now)` —
  역·요일별 시간표를 24시간 캐시(개정은 드묾), 운행일 계산(새벽 3시 컷오프), 오늘 남은
  출발이 없으면 다음 운행일 첫차(`firstOfDay: true`). 일요일=휴일 시간표로 처리하며
  **법정공휴일 감지는 미지원**(그날은 평일/토요일 표가 잘못 적용될 수 있음 — 알려진 한계).
- `TrainsService.build`: 열차가 **없는 방향에만** `nextSchedule`을 붙인다. 시간표 조회
  실패는 본 응답을 막지 않는다(null + 경고 로그).
- **프론트**: 빈 방향에서 `다음 열차 23:47 출발 (시간표 기준)` /
  운행 종료 시 `운행 종료 — 첫차 05:40 (시간표 기준)`. 조회 실패면 기존 문구 유지.
- "(시간표 기준)"을 명시해 실시간이 아님을 정직하게 알린다.

## 호출량

역·요일당 2회(방향 축 2개), 24시간 캐시 → 하루 최대 ~2회. 무시할 수준.

## 테스트

백엔드: 시간표 매퍼(실캡처 픽스처 250행), 서비스(낮/자정직후/막차후 첫차/토요일 전환/캐시),
클라이언트 봉투, trains.service 연결. 프론트: 빈 상태 3분기. 전체 127+5(백) / 124(프론트) 통과.
