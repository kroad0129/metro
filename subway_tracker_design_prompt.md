# 지하철 실시간 위치 시각화 서비스 설계 프롬프트

## 1. 역할

너는 개인용 웹 서비스의 제품 설계와 기술 설계를 돕는 시니어 풀스택 개발자다.

아래 프로젝트의 목적과 제약조건을 바탕으로, MVP 범위를 과도하게 확장하지 말고 실제 개발을 시작할 수 있는 수준의 설계를 작성해라.

설계 시 다음 원칙을 따른다.

- MVP의 핵심은 기능의 완성도가 아니라 **실제 지하철 상황을 화면에서 확인할 수 있는지 검증하는 것**이다.
- 위젯, 사용자 행동 추천, 정교한 애니메이션, 화려한 디자인 등은 초기 설계에서 제외한다.
- 초기에는 서울 지하철 9호선만 지원하되, 이후 다른 노선으로 확장할 수 있는 구조를 고려한다.
- 개인 사용을 목적으로 하며, 친구 몇 명에게 공유할 수 있는 정도의 규모를 가정한다.
- 대규모 트래픽, 결제, 관리자 페이지, 회원관리 시스템은 고려하지 않는다.
- 개발자가 이해하기 어려운 복잡한 구조보다 단순하고 명확한 구조를 우선한다.

---

## 2. 프로젝트 개요

### 프로젝트 목적

사용자가 집이나 현재 위치에서 출발하기 전에, 자주 이용하는 지하철역으로 다가오는 열차가 현재 어느 역 또는 어느 역 사이에 있는지와 도착 예정 시간을 확인하기 위한 서비스다.

사용자는 이 정보를 보고 다음과 같은 판단을 한다.

- 지금 바로 출발해야 하는지
- 뛰어가야 하는지
- 평소 속도로 걸어가도 되는지
- 열차가 오래 남았으므로 잠시 다른 일을 해도 되는지

단, MVP에서는 이러한 행동을 서비스가 직접 추천하지 않는다.

서비스는 사용자가 판단하는 데 필요한 다음 정보만 제공한다.

- 열차의 현재 위치
- 열차의 이동 방향
- 선택한 역까지의 도착 예정 시간

### 한 문장 정의

> 선택한 역으로 다가오는 양방향 열차의 현재 위치와 도착 예정 시간을 간단한 노선 형태로 보여주는 개인용 지하철 상황 확인 서비스

## 3. 사용자 및 운영 범위

### 주요 사용자

- 서비스 개발자 본인
- 필요할 경우 링크를 공유받은 소수의 친구

### 운영 규모

- 개인 프로젝트
- 소규모 사용
- 별도의 회원가입 및 로그인 없음
- 사용자별 데이터 저장 없음
- 관리자 페이지 없음
- 결제 및 광고 없음

---

## 4. 최종 목표와 현재 개발 단계

### 최종 목표

스마트폰에서 빠르게 확인할 수 있는 앱 또는 홈 화면 위젯으로 발전시키는 것이다.

### 현재 단계

최종 위젯을 바로 개발하지 않고, 우선 모바일 브라우저에서 사용할 수 있는 반응형 웹서비스로 MVP를 구현한다.

현재 단계에서는 다음을 검증한다.

> 실시간 또는 준실시간 지하철 데이터를 간단한 노선 UI에 표시했을 때, 사용자가 열차 상황을 빠르게 이해할 수 있는가?

PWA 적용 여부는 MVP 이후 검토한다.

## 5. MVP 범위

### 필수 기능

1. 9호선 역 목록을 제공한다.
2. 사용자가 기준 역을 선택할 수 있다.
3. 선택한 역을 기준으로 양방향 열차 정보를 표시한다.
4. 각 방향에서 선택한 역으로 다가오는 열차의 현재 위치를 표시한다.
5. 현재 위치는 역 또는 역 사이 위치로 표현한다.
6. 선택한 역까지의 도착 예정 시간을 숫자로 표시한다.
7. 일정 주기로 데이터를 다시 조회하고 화면을 갱신한다.
8. 데이터의 마지막 갱신 시간을 표시한다.
9. API 오류 또는 데이터 없음 상태를 사용자에게 구분해서 보여준다.

### MVP에서 제외할 기능

- 로그인 및 회원가입
- 사용자 계정
- 즐겨찾기
- 최근 검색
- 푸시 알림
- 홈 화면 위젯
- 네이티브 앱
- 사용자의 집과 역 사이 도보 시간 저장
- 뛰기, 걷기 등의 행동 추천
- 사용자 위치 또는 GPS
- 지도
- 실제 거리 비율에 따른 역 간격
- 정교한 열차 아이콘
- 복잡한 애니메이션
- 확대 및 축소
- 노선 전체를 실제 지하철 노선도처럼 그리는 기능
- 데이터베이스
- 관리자 페이지
- 대규모 트래픽 대응
- WebSocket
- Redis

## 6. 초기 지원 범위

### 노선

- 서울 지하철 9호선만 지원한다.
- 이후 다른 노선을 추가할 수 있도록 노선 데이터와 화면 로직을 특정 노선에 지나치게 종속시키지 않는다.

### 기준 예시 역

- 증미역을 주요 테스트 역으로 사용한다.
- 증미역을 기준으로 양방향 열차가 표시되어야 한다.

### 역 간격

- 모든 역 간격은 동일하게 표시한다.
- 실제 역 사이 거리나 실제 지리적 위치는 반영하지 않는다.
- 사용자는 도착 예정 시간을 숫자로 함께 확인하므로 정확한 거리 비율은 필요하지 않다.

## 7. UI 원칙

### 기본 형태

실제 지하철 노선도를 정교하게 그리지 않는다.

HTML과 CSS를 이용해 다음 정도의 단순한 형태로 구현한다.

```text
가양 ●────────● 증미 ────────● 등촌
          ● 열차
          2분
```

또는 방향별로 다음처럼 표시한다.

```text
개화 방면

마곡나루 ─ 신방화 ─ 공항시장 ─ 김포공항
              ●
             3분

중앙보훈병원 방면

염창 ─ 등촌 ─ 증미 ─ 가양
       ●
      2분
```

### 열차 표현

- 초기에는 열차 아이콘 대신 점 또는 원을 사용한다.
- 점은 열차 위치에 따라 CSS의 `left` 또는 `transform` 값을 변경해 이동시킨다.
- API 갱신 시 위치가 변경되면 화면에서도 위치를 갱신한다.
- CSS `transition`을 사용할 수 있지만, 정교한 애니메이션 구현은 하지 않는다.
- 열차 아이콘은 MVP 이후 개선사항으로 남긴다.

### 모바일 우선

- 스마트폰 세로 화면을 우선으로 설계한다.
- 데스크톱에서도 사용할 수 있어야 하지만 모바일 UI를 기준으로 한다.
- 별도 디자인 시스템이나 UI 라이브러리는 필수가 아니다.
- 일반 CSS 또는 CSS Module 수준으로 구현한다.

## 9. 권장 시스템 구성

```text
스마트폰 브라우저
        ↓ HTTPS
개인 도메인
        ↓
nginx
   ├── 프론트엔드 정적 파일 제공
   └── /api 요청을 NestJS로 프록시
                    ↓
               서울시 OpenAPI
```

### nginx 역할

- Vite로 빌드한 프론트엔드 정적 파일 제공
- `/api` 요청을 NestJS 백엔드로 리버스 프록시
- 도메인 연결
- HTTPS 적용
- 외부에는 80, 443 포트만 노출

### Docker Compose 구성

초기에는 다음 두 서비스로 구성할 수 있다.

```text
docker-compose
├── backend
│   └── NestJS
└── nginx
    ├── React 빌드 결과 제공
    └── backend 리버스 프록시
```

프론트엔드는 Docker 빌드 과정에서 Vite로 빌드하고, 생성된 `dist` 파일을 nginx 이미지에 복사하는 멀티 스테이지 빌드를 사용할 수 있다.

## 11. 정적 노선 데이터

9호선 역 목록은 JSON 파일로 저장한다.

외부 공공데이터에서 한 번 수집해 JSON으로 변환하거나, 초기에는 직접 작성해도 된다.

이 데이터는 실시간으로 매번 조회하지 않는다.

예시:

```json
{
  "lineId": "9",
  "lineName": "서울 지하철 9호선",
  "stations": [
    {
      "stationId": "station-001",
      "name": "개화",
      "order": 1
    },
    {
      "stationId": "station-002",
      "name": "김포공항",
      "order": 2
    },
    {
      "stationId": "station-003",
      "name": "공항시장",
      "order": 3
    }
  ]
}
```

### 확장성 원칙

- 역 순서는 `order`로 관리한다.
- 방향은 역 순서의 증가 또는 감소로 판단할 수 있게 한다.
- 노선별 JSON 파일을 추가할 수 있는 구조로 만든다.
- 프론트엔드에 9호선 역 이름을 직접 하드코딩하지 않는다.

## 12. 백엔드 API 초안

### 역 목록 조회

```http
GET /api/lines/9/stations
```

응답 예시:

```json
{
  "lineId": "9",
  "lineName": "서울 지하철 9호선",
  "stations": [
    {
      "stationId": "0922",
      "name": "증미",
      "order": 10
    }
  ]
}
```

### 특정 역 열차 상황 조회

```http
GET /api/lines/9/stations/{stationId}/trains
```

응답 예시:

```json
{
  "line": {
    "id": "9",
    "name": "서울 지하철 9호선"
  },
  "station": {
    "id": "0922",
    "name": "증미",
    "order": 10
  },
  "directions": [
    {
      "directionId": "UP",
      "directionName": "개화 방면",
      "trains": [
        {
          "trainId": "9134",
          "currentStation": {
            "id": "0923",
            "name": "등촌",
            "order": 11
          },
          "nextStation": {
            "id": "0922",
            "name": "증미",
            "order": 10
          },
          "remainingSeconds": 125,
          "status": "APPROACHING",
          "positionRatio": 0.45
        }
      ]
    },
    {
      "directionId": "DOWN",
      "directionName": "중앙보훈병원 방면",
      "trains": []
    }
  ],
  "updatedAt": "2026-07-22T14:00:00+09:00"
}
```

### 응답 설계 원칙

- 서울시 API 원본 응답을 프론트엔드에 그대로 전달하지 않는다.
- 외부 API 필드명을 내부 도메인 필드명으로 변환한다.
- 프론트엔드는 외부 API 형식을 알 필요가 없어야 한다.
- 외부 API 변경 시 백엔드만 수정할 수 있도록 한다.

## 13. 열차 위치 계산

### 기본 원칙

열차 위치가 특정 역으로 제공되면 해당 역 위치에 표시한다.

열차가 두 역 사이에 있다고 판단할 수 있는 데이터가 있다면 두 역 사이에 표시한다.

### 단순 위치 비율

역 A와 역 B 사이에서 열차의 위치를 나타내기 위해 `positionRatio`를 사용할 수 있다.

```text
0.0 = 이전 역 위치
0.5 = 두 역의 중간
1.0 = 다음 역 위치
```

정확한 계산이 어렵거나 API 데이터가 부족하면 MVP에서는 다음과 같이 단순화해도 된다.

- 이전 역 출발 상태: 0.25
- 역 사이 이동 상태: 0.5
- 다음 역 진입 상태: 0.75
- 역 도착 상태: 1.0

초기 목표는 GPS 수준의 정확도가 아니라, 사용자가 열차가 어느 구간에 있는지 이해할 수 있게 하는 것이다.

## 14. 데이터 갱신 및 캐시

### 호출 주기

정확한 호출 주기는 초기 설계에서 고정하지 않는다.

다음과 같이 설정값으로 분리한다.

```env
SUBWAY_POLLING_INTERVAL_MS=15000
SUBWAY_CACHE_TTL_MS=10000
```

개발 및 테스트 과정에서 서울시 API 호출 제한과 실제 사용성을 확인해 조정한다.

### 캐시 원칙

- 같은 9호선 실시간 위치 데이터를 짧은 시간 내 반복 호출하지 않는다.
- 여러 프론트 요청이 들어와도 캐시된 데이터를 재사용할 수 있게 한다.
- MVP에서는 Redis 없이 NestJS 프로세스의 메모리 캐시를 사용한다.
- 서버 재시작 시 캐시가 사라져도 문제가 없어야 한다.

## 15. 환경변수

SEOUL_SUBWAY_REALTIME_BASE_URL=

### 보안 원칙

- 서울시 API 키를 프론트엔드에 포함하지 않는다.
- API 키를 Git 저장소에 커밋하지 않는다.
- 실제 `.env` 파일은 서버에서만 관리한다.
- 프론트엔드는 NestJS API만 호출한다.

## 16. 예외 처리

최소한 다음 상태를 구분한다.

### 로딩

- 첫 데이터를 불러오는 중임을 표시한다.

### 데이터 없음

- 해당 방향에서 접근 중인 열차가 없음을 표시한다.

### 외부 API 오류

- 현재 데이터를 가져오지 못했음을 표시한다.
- 가능하다면 마지막 정상 데이터와 마지막 갱신 시간을 유지한다.

### 호출 제한

- 서울시 API 호출 제한 오류를 백엔드에서 감지한다.
- 프론트엔드에는 내부 오류 형식으로 변환해 전달한다.

### 잘못된 역

- 지원하지 않는 역 ID 요청 시 404 또는 적절한 오류를 반환한다.

## 17. MVP 성공 기준

다음 조건을 만족하면 MVP가 성공한 것으로 본다.

1. 스마트폰에서 서비스에 접속할 수 있다.
2. 9호선 역을 선택할 수 있다.
3. 선택한 역 기준 양방향 열차 정보를 볼 수 있다.
4. 열차가 어느 역 또는 어느 구간에 있는지 알 수 있다.
5. 선택한 역까지 남은 시간을 숫자로 확인할 수 있다.
6. 데이터가 일정 주기로 갱신된다.
7. 실제 외출 전에 사용해 봤을 때 열차 상황을 판단하는 데 도움이 된다.

UI가 완성도 높거나 열차가 부드럽게 움직이는지는 MVP 성공 조건에 포함하지 않는다.

8. 9호선 JSON 데이터 구조

17. 테스트 항목
18. MVP 이후 확장 포인트

### 작성 시 주의사항

- MVP에 필요하지 않은 기능을 임의로 추가하지 마라.
- 복잡한 아키텍처를 제안하지 마라.
- Redis, Kafka, Kubernetes, 마이크로서비스는 사용하지 마라.
- 데이터베이스는 사용하지 마라.
- WebSocket 대신 일반 HTTP 폴링을 사용한다.
- 프론트엔드 UI는 HTML과 CSS 기반으로 단순하게 구현한다.
- 초기 구현은 서울 지하철 9호선에 집중한다.
- 이후 다른 노선을 추가할 수 있는 정도의 확장성만 고려한다.
- 각 기술 선택의 이유를 짧게 설명한다.
- 구현 예시는 실제 코드 작성에 바로 활용할 수 있을 정도로 구체적으로 작성한다.
- 불확실한 서울시 API 필드나 동작은 추측하지 말고 확인이 필요한 항목으로 구분한다.
